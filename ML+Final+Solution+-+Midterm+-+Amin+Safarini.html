Which metric to use¶

  • Recall: from the actual defaults, how many did you capture correctly
  • Precision: from the predicted defaults, how many were actually correct

Recall would be important to the bank to make sure that they don't lose money. You want to correctly capture the loans that will default.

Tuning down the max_depth will allow you to obtain a higher recall at the cost of precision

I can quite easily improve my precision by maximizing max_depth, however this would be at the cost of a lower recall

but we will use GRID SEARCH below to discover optimal results as well¶

in conlusion the Grid search slightly helps with the precision but doens't give a higher recall compared to my manual hyperparameter tuning.¶

  • Recall: 57%
  • Precision: 76%
  • F1: 65%

1) define objective -> loan default prediction for clients applying for their first loan. We will predict based on loan status (our label) and prior to the loan being issued.

  • bad current and finished loans grouped together (B & D)
  • good current and finished loans grouped together (A & C)
  • Decide on the metric to use and which benefits our business. We think recall on bad loans is our best metric but also needs to be balanced with a good f1 score

2) load data from SQL 3) clean data -> make sure that there are no object types to work with sklean, drop any useless columns data.

  • one hot encode categorical data 4) explore your data -> summary stats, visualizations. look at distribution of labels (imbalanced) 5) feature engineer -> experiment with creating features that will potentially benefit the model and have a high importance to our model 6) ML modeling:
  • fill/impute gaps (missing values) at this stage to be safe and prevent data leakage
  • we have a classificatio problem -> we used Random Forests since it gave us the best results
  • we have imbalanced data -> need to oversample. Best practice to use smote in order to prevent magnifying noise
  • no scaling required for Random Forest. Needs to be scaled for other models such as logestic regression
  • Always split data into training and test sets. Since the data is imbalanced, we were hesitant to use a higher % for testing. Stick to 20-30%.
  • Use grid search for hyperparameter tuning and cross validation -> set scoring to recall from trained classification report
  • plot feature importance, re-iterate if necessary to improve model (drop/add/modify features). Be careful to observe any cases where we may have overfitting
  • potentially use a pipeline, probably not requied for a small dataset like hours
  • repeat

Challenges:

  • hyperparameter tuning

Conclusion:

  • random forest gave best results compared to KNN and logestic regression
  • higher scores for good loans, lower scores for bad loans (since data is imbalanced)
  • however oversampling improves the scores for bad loans
  • min balance before loan, amount borrowed, payments and the year the account was created were some of the most important features
